# Phase 8.5: Workspace Onboarding + Repo Indexing - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Mandatory onboarding for existing Git repositories. The user registers a project from a local workspace path or a remote repo URL, the system validates and indexes the repo, generates initial vault documentation via GSD InitRunner (AI-powered, brownfield mode), and establishes the safe execution model: host repo is the source of truth; each run uses an ephemeral workspace clone; promotion is explicit.

</domain>

<decisions>
## Implementation Decisions

### Entry point priority

- **D-01:** CLI is the primary entry point for onboarding. A separate `@bridge-ai/cli` package (binary) is the canonical way to trigger `bridge init`.
- **D-02:** CLI uses the **lazy daemon + proxy** pattern: on `bridge init`, the CLI checks if the daemon is running; if not, it starts it automatically before proxying the command via HTTP to `localhost`.
- **D-03:** Installation via `npm install -g @bridge-ai/cli`. No npm publish for MVP — use local `npm link` / `bun link` for development.
- **D-04:** Telegram is the secondary entry point (primary for VPS operators). It calls the same underlying daemon logic as the CLI.
- **D-05:** REST API (`POST /workspaces/init`) is **out of scope for Phase 8.5 MVP**. The existing research recommendation of REST-first is superseded by CLI-first. REST endpoint may be added in a later phase.

### Target deployment model

- **D-06:** Primary target is **local development** — user installs bridge-ai on their own machine, runs the daemon locally, uses CLI in the same terminal. VPS deployment (with Telegram as primary interface) is a secondary use case.
- **D-07:** This changes the design constraint: the CLI can assume `localhost` daemon access. No remote daemon addressing needed for Phase 8.5.

### Indexing flow

- **D-08:** `bridge init --workspace <path>` (or `--repo <url> --workspace <path>`) runs two sequential stages in the same command:
  1. **Extraction (offline, no AI):** git remote + branch info, package.json / Cargo.toml / go.mod / pyproject.toml detection, directory tree, entrypoints detection, existing docs (README, docs/).
  2. **AI doc generation (GSD InitRunner, brownfield mode):** extraction snapshot is passed as context to InitRunner. AI generates project.md, architecture.md, stack.md, decisions.md, runbook.md. Uses the existing GSD pattern already implemented in `packages/gsd-sdk`.
- **D-09:** There is **no separate enrichment step**. Extraction and AI doc generation are part of the same `bridge init` flow. The user gets fully documented vault output from a single command.
- **D-10:** Extraction is the foundation that makes the AI useful for brownfield repos — without it, BrainService would have no grounded context. With it, InitRunner operates on real data instead of guessing.

### GSD InitRunner adaptation

- **D-11:** InitRunner already implements the 7-step GSD init workflow (setup → config → PROJECT.md → parallel research × 4 → synthesis → requirements → roadmap). For Phase 8.5, it needs to accept an `extractionSnapshot` as input so the research steps have real repo context rather than starting from user description alone.
- **D-12:** The four parallel research sessions (STACK, FEATURES, ARCHITECTURE, PITFALLS) receive the extraction snapshot as additional context. This is a brownfield adaptation of the existing greenfield flow.

### Vault docs output

- **D-13:** Initial vault docs generated per onboarded project: `project.md`, `architecture.md`, `stack.md`, `decisions.md`, `runbook.md`. Synced via existing `ObsidianSyncService`.
- **D-14:** Docs live in `volumes/obsidian/projects/<slug>/` — same structure established in Phase 6.

### Claude's Discretion

- Exact schema for `workspace_snapshots` PostgreSQL entity (JSONB payload structure, indexes)
- `simple-git` API call selection for remote/branch detection
- Ephemeral workspace clone strategy: `git clone --local` depth and location (`volumes/runs/<runId>/`)
- Promotion strategies implementation detail (patch / cherry-pick / push)
- How the CLI binary detects if the daemon is alive (health endpoint ping) and how it starts it (child_process spawn)
- Incremental indexing trigger mechanism for Phase 8.5 MVP (on-demand vs automatic)

</decisions>

<specifics>
## Specific Ideas

- "lazy daemon + CLI como proxy" — user's exact phrase for the CLI/daemon relationship. CLI is the user-facing tool; daemon is infrastructure the user shouldn't need to think about.
- Primary use case is local machine development, not VPS. This shifts the mental model from "remote operator via Telegram" to "developer using bridge-ai as a local tool."
- The GSD pattern is the core value: `bridge init` on a brownfield repo should produce the same quality of documentation as starting a greenfield project with GSD. Extraction makes that possible.

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ObsidianSyncService`: already writes vault docs per project; `syncProject()` and `ensureVaultStructure()` can be called from the onboarding flow directly
- `KsmService.createSecret()` / `getSecret()`: use for SSH key passphrase or HTTPS PAT storage per project
- `ProjectService.create()`: called at end of onboarding to register the project in PostgreSQL
- `GSD InitRunner`: core AI doc generation engine; needs brownfield adaptation (extractionSnapshot input)
- `node:fs/promises` + `node:path`: already used throughout — no extra deps for directory tree walking

### Established Patterns
- NestJS module with `@Injectable()` services, `@InjectRepository()` for TypeORM, `@Module()` for wiring — follow existing modules as templates
- TypeORM migration: new file with timestamp > 1711843200000 in `apps/api/src/migrations/`
- BullMQ queue pattern (if async indexing is needed): follow `execution.worker.ts`

### Integration Points
- New `WorkspaceModule` at `packages/nest-core/src/module/workspace/` — registered in `AppModule`
- `PipelineService.executeProject()` must swap `WorkspaceService.provisionWorkspace()` → `EphemeralWorkspaceService.cloneForRun()` so runs use isolated clones
- `DockerService.createContainer()` receives ephemeral workspace path (not host path) — existing mount logic unchanged, just different path source
- CLI package at `packages/cli/` or `apps/cli/` — new package, proxies to daemon via HTTP (`/health` for alive check, `/workspaces/init` for onboarding command)

</code_context>

<deferred>
## Deferred Ideas

- REST API endpoint `POST /workspaces/init` — explicitly out of scope for Phase 8.5 MVP; add in later phase if needed
- npm publish of `@bridge-ai/cli` to npm registry — deferred; use `npm link` locally for now
- Remote daemon addressing (non-localhost) — not needed for local-first deployment model
- Automatic incremental indexing triggers (webhook, filesystem watch) — Phase 8.5 delivers on-demand only; automation is a follow-up
- Telegram wrapper for `bridge init` — per D-04, CLI is the primary entry point for Phase 8.5; Telegram onboarding command deferred to Phase 8.6 (Telegram refactor / deterministic protocol)

</deferred>
