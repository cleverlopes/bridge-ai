## Summary

Phase 8.5 introduces the **safe workspace model** that all downstream phases (Loop Engine, Policy Engine) depend on. Three interlocking concerns must be implemented together: (1) repository onboarding that validates and indexes an existing Git repo, (2) per-run ephemeral workspace cloning that isolates all agent writes, and (3) the promotion contract that makes changes from a run visible in the host repo only by explicit operator action.

The existing `WorkspaceService` in `packages/nest-core/src/module/pipeline/` provisions a `.planning/` directory inside a stable per-project workspace. This model is not the target safe model — it allows the agent to write freely to that path. Phase 8.5 introduces a new `WorkspaceModule` at `packages/nest-core/src/module/workspace/` that replaces and supersedes the pipeline-local `WorkspaceService` for run provisioning. The existing `WorkspaceService` can be migrated into the new module or kept as-is and superseded for new run paths.

Git operations should use `simple-git` v3.33.0 (latest stable, pure JavaScript, no native dependencies, excellent TypeScript types, widely adopted in Node.js ecosystems). The `workspace_snapshots` table needs a JSONB `indexPayload` column, FK to `projects`, and fields for HEAD SHA, remote URL, base branch, and snapshot timestamp. A second migration file (timestamp > 1711843200000) handles this schema addition cleanly.

**Primary recommendation:** Create `WorkspaceModule` with four services (`WorkspaceOnboardingService`, `RepoIndexerService`, `EphemeralWorkspaceService`, `PromotionService`), add `workspace_snapshots` entity + migration, surface `gateway init` as a REST endpoint (`POST /workspaces/init`) that the Telegram command wrapper and the future CLI both call.

### Pattern 1: WorkspaceOnboardingService — the `gateway init` flow

**What:** Validates that a given path is a Git repo (or clones a URL into a path), detects repo metadata, validates credentials, persists to DB, triggers bootstrap indexing.

**When to use:** Called once per project registration. Idempotent — re-running for the same project refreshes the snapshot.

```typescript
// Source: simple-git 3.x API — https://github.com/steveukx/git-js
import simpleGit, { SimpleGit, CheckRepoActions } from 'simple-git';

async validateRepo(workspacePath: string): Promise<RepoInfo> {
  const git: SimpleGit = simpleGit(workspacePath);
  const isRepo = await git.checkIsRepo(CheckRepoActions.IS_REPO_ROOT);
  if (!isRepo) throw new BadRequestException(`${workspacePath} is not a git repository root`);

  const remotes = await git.getRemotes(true);
  const mainRemote = remotes.find(r => r.name === 'origin') ?? remotes[0] ?? null;

  const status = await git.status();
  const headSha = await git.revparse(['HEAD']);

  // Detect base branch: try symbolic-ref then fallback to 'main'/'master'
  let baseBranch = 'main';
  try {
    const ref = await git.raw(['symbolic-ref', '--short', 'refs/remotes/origin/HEAD']);
    baseBranch = ref.trim().replace('origin/', '');
  } catch {
    // not fatal — use 'main' as default
  }

  return {
    remoteUrl: mainRemote?.refs.fetch ?? null,
    remoteName: mainRemote?.name ?? null,
    baseBranch,
    currentBranch: status.current ?? 'HEAD',
    isDirty: status.files.length > 0,
    headSha: headSha.trim(),
  };
}
```

### Pattern 2: Auth Validation (ONBOARD-03)

**What:** For HTTPS: attempt a `git ls-remote` (read test) and a `git push --dry-run` (write test). For SSH: test with `git ls-remote` using the configured SSH key. Credentials stored in KSM, never in env.

**Operationally, "validate read/write" means:**
- **Read test:** `git ls-remote <url>` returns without error — confirms authentication + network access + repo existence
- **Write test:** `git push --dry-run origin HEAD` from the host workspace — confirms push access without creating a remote commit

```typescript
// Read validation via simple-git
const testGit = simpleGit(workspacePath);
await testGit.listRemote(['--heads', remoteUrl]); // throws on auth failure

// Write validation — dry run push
await testGit.push(['--dry-run', 'origin', 'HEAD']); // throws on permission denied
```

**SSH credential storage contract:**
```
KSM secret name: workspace-ssh-key-<projectId>
KSM scope: 'project'
KSM scopeId: projectId
Value: PEM private key string (base64-encoded)
```

**HTTPS credential storage contract:**
```
KSM secret name: workspace-https-pat-<projectId>
KSM scope: 'project'
KSM scopeId: projectId
Value: personal access token string
```

For SSH: write the key to a temp file, set `GIT_SSH_COMMAND` env variable for the git operation, clean up immediately after. Never persist key material to disk beyond the call scope.

### Pattern 3: EphemeralWorkspaceService — per-run clone

**What:** Creates a full isolated clone of the registered host workspace into `volumes/runs/<runId>/`. Container mounts only this path. Path is deleted after run completes.

**Decision: `git clone --local` vs `cp -r`:**
- `git clone --local <src> <dst>` — creates a proper `.git/` directory, hardlinks pack files (low disk cost on same filesystem), isolates object store, standard git operations work
- `cp -r` — simpler but copies full history, larger disk footprint, no git semantics

Use `git clone --local` as the primary strategy. Falls back to `cp -r` if the source is a shallow clone without pack files.

```typescript
// Source: simple-git clone API
async cloneForRun(hostWorkspacePath: string, runId: string): Promise<string> {
  const runPath = join(RUNS_BASE, runId);
  await mkdir(RUNS_BASE, { recursive: true });

  const git = simpleGit();
  await git.clone(hostWorkspacePath, runPath, ['--local', '--no-hardlinks']);
  // --no-hardlinks forces file copies, safer for concurrent runs on same host

  this.logger.log(`Ephemeral workspace cloned for run ${runId} at ${runPath}`);
  return runPath;
}

async cleanupRun(runId: string): Promise<void> {
  const runPath = join(RUNS_BASE, runId);
  await rm(runPath, { recursive: true, force: true });
  this.logger.log(`Ephemeral workspace cleaned for run ${runId}`);
}
```

**Integration with DockerService:** `PipelineService` must be updated to call `EphemeralWorkspaceService.cloneForRun()` instead of `WorkspaceService.provisionWorkspace()` when a registered workspace exists. The `workspacePath` passed to `DockerService.createContainer()` becomes the ephemeral run path.

### Pattern 4: MVP Indexing Extraction

**What the RepoIndexerService extracts (full bootstrap):**

| Category | What | How |
|----------|------|-----|
| File tree | Top-level dirs + depth-2 structure | `readdir` recursive with depth limit |
| Manifests | `package.json`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `pom.xml`, `build.gradle` | Glob at root + common dirs |
| Entrypoints | `main.ts`, `index.ts`, `app.ts`, `main.py`, `__main__.py`, `main.go` | Pattern match on file tree |
| Test files | Files matching `*.spec.ts`, `*.test.*`, `test/`, `tests/`, `__tests__/` | Pattern scan |
| Docs | `README.md`, `CHANGELOG.md`, `docs/` directory, `.planning/` contents | Direct path check |
| Remote + branch info | From `RepoInfo` struct produced during onboarding | Stored from git metadata |

**Storage format (JSONB `indexPayload`):**
```typescript
interface IndexPayload {
  tree: string[];              // relative paths, depth-limited
  manifests: ManifestEntry[];  // { path, type, content (partial) }
  entrypoints: string[];       // relative paths
  testPaths: string[];         // relative paths
  docPaths: string[];          // relative paths
  remoteUrl: string | null;
  remoteName: string | null;
  baseBranch: string;
  currentBranch: string;
  headSha: string;
  indexedAt: string;           // ISO timestamp
}
```

**Incremental sync trigger:** On every run start (compare stored `headSha` with `git rev-parse HEAD`); also exposed as `POST /workspaces/:projectId/sync` for manual trigger. Cron (every 15 min) is an option but deferred to post-8.5 — manual trigger + run-start trigger is sufficient for MVP.

### Pattern 5: WorkspaceSnapshot Entity (ONBOARD-05)

**Minimal schema:**
```sql
CREATE TABLE "workspace_snapshots" (
  "id" uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  "projectId" uuid NOT NULL REFERENCES "projects"("id") ON DELETE CASCADE,
  "workspacePath" text NOT NULL,           -- absolute path to host repo
  "remoteUrl" text,                        -- git remote URL (may be null for local-only)
  "remoteName" varchar(50),               -- typically 'origin'
  "baseBranch" varchar(255) NOT NULL DEFAULT 'main',
  "currentBranch" varchar(255) NOT NULL,
  "headSha" varchar(40) NOT NULL,
  "isDirty" boolean NOT NULL DEFAULT false,
  "indexPayload" jsonb NOT NULL DEFAULT '{}',
  "indexedAt" TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  "createdAt" TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  "updatedAt" TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT now(),
  CONSTRAINT "UQ_workspace_snapshots_projectId" UNIQUE ("projectId")
  -- one snapshot per project; upsert on re-onboard
);
CREATE INDEX "IDX_workspace_snapshots_projectId" ON "workspace_snapshots" ("projectId");
CREATE INDEX "IDX_workspace_snapshots_headSha" ON "workspace_snapshots" ("headSha");
```

Note: The `UNIQUE` constraint on `projectId` enforces one active snapshot per project. Re-running `gateway init` updates (upserts) the existing row.

### Pattern 6: Vault Docs Generation (ONBOARD-04)

The five initial docs are generated by `WorkspaceOnboardingService` after successful indexing and written via `ObsidianSyncService` patterns (direct `fs/promises` writes to vault path):

| File | Location | Content source |
|------|----------|----------------|
| `project.md` | `volumes/obsidian/projects/<slug>/project.md` | Project name, description, stack from index manifest |
| `architecture.md` | `volumes/obsidian/projects/<slug>/architecture.md` | Inferred from manifest + entrypoints + tree |
| `stack.md` | `volumes/obsidian/projects/<slug>/stack.md` | Parsed from manifest files (package.json deps etc.) |
| `decisions.md` | `volumes/obsidian/projects/<slug>/decisions.md` | Empty ADR scaffold with frontmatter |
| `runbook.md` | `volumes/obsidian/projects/<slug>/runbook.md` | Stub with repo URL, branches, clone instructions |

These files are generated from the index payload directly (no AI call needed for MVP stubs). An optional AI-enhancement pass using `BrainService.generate()` can enrich them but is not required for ONBOARD-04 compliance.

### Pattern 7: PromotionService (ONBOARD-10)

Three explicit promotion paths, none of which involve the daemon writing directly to the host repo:

```typescript
type PromotionStrategy = 'patch' | 'cherry-pick' | 'push';

// Strategy 1: patch — generate a diff from ephemeral workspace, apply to host
async promoteViaPatch(runId: string, hostPath: string): Promise<void>

// Strategy 2: cherry-pick — cherry-pick specific commit SHAs from ephemeral into host
async promoteViaCherryPick(runId: string, hostPath: string, commitShas: string[]): Promise<void>

// Strategy 3: push — push the run branch from ephemeral workspace directly to remote
async promoteViaPush(runId: string, remoteName: string, branchName: string): Promise<void>
```

**Safety invariant:** `PromotionService` takes the `hostPath` as a parameter (never derives it itself) and the caller must have explicitly loaded it from the registered `workspace_snapshots` record. The daemon does NOT hold a global reference to any host workspace path.

### Pattern 8: Telegram Command Wrapper (`/init` command)

Since Phase 8.6 (Telegram refactor) is not yet implemented, Phase 8.5 should surface `gateway init` as:
1. **REST endpoint** `POST /workspaces/init` with body `{ workspacePath, repoUrl?, projectName, slug }` — primary surface for CLI and programmatic use
2. **Telegram command** `/init <workspace_path>` added to the existing `TelegramBotService` — thin wrapper that calls the same service

This keeps the Telegram surface minimal and lets Phase 8.6 refactor the command properly without touching the core onboarding logic.

### Anti-Patterns to Avoid

- **Storing SSH private keys in `.env` or as plain text in the database:** Always use `KsmService.createSecret()` with `scope: 'project'`.
- **Mounting the host repo with write permissions in Docker:** The `Binds` array in `DockerService.createContainer()` must always receive the ephemeral run path, never the `workspace_snapshots.workspacePath`.
- **Global git config mutation during auth testing:** Use `GIT_SSH_COMMAND` env var scoped to the git process, not `git config --global`.
- **Recursive tree walk without depth limit:** Repos can have thousands of files. Cap depth at 3 and file count at 2000 for the tree index to keep indexPayload bounded.
- **Running full bootstrap on every run start:** Only bootstrap on first onboarding or explicit re-index. Use headSha comparison for incremental checks on subsequent runs.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Git repository operations (clone, status, remote, log, diff, apply) | Custom `child_process` git wrapper | `simple-git` v3 | Handles output parsing, error codes, streaming, SSH config, concurrent operations correctly |
| Recursive directory tree | Manual `readdir` loop with recursion | `node:fs/promises` `readdir` with `{ recursive: true }` (Node 18.17+) | Already available; tested built-in |
| Glob pattern matching for manifest/entrypoint detection | Custom regex tree walker | `node:fs/promises` + simple pattern matching on paths | No extra dep needed at MVP scale |
| Credential storage | `.env` entries or plain DB columns | `KsmService.createSecret()` | AES-256-GCM already implemented; audit trail free |
| TypeORM migration | Manual SQL scripts | TypeORM migration class (timestamp file) | Consistent with existing schema; runs on startup |

**Key insight:** The hard problems in git tooling are SSH auth, stream demux, exit code mapping, and concurrent process management. `simple-git` handles all of these correctly. The only reason to reach for the raw CLI would be a `simple-git` API gap, which is unlikely at v3.33.

## Common Pitfalls

### Pitfall 1: SSH Key Handling in Ephemeral Context

**What goes wrong:** Writing the SSH private key to `/tmp/id_rsa`, setting `GIT_SSH_COMMAND`, and forgetting to delete it if the operation throws.

**Why it happens:** Async error paths skip cleanup.

**How to avoid:** Use a `try/finally` block around any temp file write. Consider using `node:os.tmpdir()` + random suffix, and always `unlink` in `finally`.

**Warning signs:** `SecretAudit` rows show repeated reads of the SSH key secret without corresponding cleanup events.

### Pitfall 2: `git clone --local` Hardlink Corruption on Concurrent Runs

**What goes wrong:** Two runs clone the same host repo simultaneously; one run modifies a hardlinked pack file that the other is reading.

**Why it happens:** `--local` without `--no-hardlinks` shares pack files.

**How to avoid:** Always pass `--no-hardlinks` to the clone command. The performance cost is acceptable (SSD copy is fast; pack files are usually < 50MB for typical repos).

**Warning signs:** Git errors like "pack file is corrupt" or "object not found" appearing only under concurrent runs.

### Pitfall 3: WorkspaceSnapshot headSha Drift

**What goes wrong:** The registered `headSha` in `workspace_snapshots` diverges from the actual repo HEAD (operator did a `git pull` on the host). The incremental sync then compares stale vs stale and incorrectly decides no re-index is needed.

**Why it happens:** `headSha` is only updated on explicit re-onboard or incremental sync trigger.

**How to avoid:** Read the live `git rev-parse HEAD` at run start, compare with stored `headSha`, and trigger sync if they differ before provisioning the ephemeral workspace.

**Warning signs:** Container sees different code than what the operator expects; vault docs are out of date.

### Pitfall 4: `indexPayload` JSONB Size Unbounded

**What goes wrong:** Indexing a large monorepo produces a `tree` array with 50,000+ entries, bloating the `indexPayload` JSONB column and slowing queries.

**Why it happens:** No depth or count limit on the tree walk.

**How to avoid:** Cap `tree` entries at 2,000 paths, depth at 3. Store a `truncated: true` flag in the payload if the limit was hit.

**Warning signs:** `workspace_snapshots` rows > 1MB; index queries become slow.

### Pitfall 5: Missing `onModuleDestroy` Cleanup for Ephemeral Workspaces

**What goes wrong:** Application restarts leave orphan run directories in `volumes/runs/`.

**Why it happens:** `rm -rf` only called in the happy path.

**How to avoid:** `EphemeralWorkspaceService.onModuleInit()` should scan `volumes/runs/` and clean up any directories whose `runId` has no corresponding active plan in the database. Alternatively, store run directory paths in `workspace_snapshots` or a `runs` audit table.

**Warning signs:** `volumes/runs/` grows unboundedly; disk usage alerts.

### Full repo status detection

```typescript
// Source: simple-git 3.x — status, getRemotes, revparse
const [remotes, status, headSha] = await Promise.all([
  git.getRemotes(true),           // verbose = true returns fetch/push URLs

const result = {
```

### TypeORM entity for WorkspaceSnapshot

```typescript
import { Entity, PrimaryGeneratedColumn, Column, CreateDateColumn,
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `git worktree` for isolation | `git clone --local --no-hardlinks` | Best practice 2022+ | Full isolation; no shared object store risk |
| Credential in `.gitconfig` | `GIT_SSH_COMMAND` env + temp key file | Node.js git tooling standard | No global config mutation; process-scoped |
| `nodegit` (libgit2 bindings) | `simple-git` (CLI wrapper) | simple-git reached parity ~v2 | No native deps; easier cross-platform deployment |
| Recursive `readdirSync` | `readdir` with `{ recursive: true }` | Node.js 18.17.0 (July 2023) | Built-in; no external dep |

**Deprecated/outdated:**
- `nodegit`: Last release 2022; native bindings fail on Node 20+; community migrated to simple-git or isomorphic-git
- `git worktree` for per-run isolation: Still valid but requires shared object store — introduces coupling between runs on the same machine
- Storing workspace paths in `Plan.workspacePath`: The existing `plans.workspacePath` column exists but is currently unused (null). Phase 8.5 introduces `workspace_snapshots.workspacePath` for the registered host path. The plans column should be repurposed or deprecated to avoid confusion.

## Open Questions

1. **Does `PipelineService` need to be updated in this phase?**
   - What we know: `PipelineService.executeProject()` currently calls `WorkspaceService.provisionWorkspace()` to get a path, then passes it to `DockerService.createContainer()`.
   - What's unclear: Should Phase 8.5 update `PipelineService` to call `EphemeralWorkspaceService.cloneForRun()` instead, or is that a Phase 8.6/10 concern?
   - Recommendation: Yes, update `PipelineService` in this phase. ONBOARD-08 and ONBOARD-09 are hard requirements. The `PipelineService` must use ephemeral paths for container mounts. Keep the change minimal — just swap the workspace provisioning call.

2. **Should the existing `WorkspaceService` in `pipeline/` be removed or kept?**
   - What we know: It provisions `.planning/` dirs in `volumes/workspaces/<projectId>/`. This is the old model.
   - What's unclear: Are there tests depending on it that would break?
   - Recommendation: Keep `WorkspaceService` as-is for the MVP `.planning/` provisioning inside ephemeral workspaces. The new `EphemeralWorkspaceService` handles the clone; `WorkspaceService` still creates `.planning/` inside that clone if needed.

3. **What is the promotion UX in this phase?**
   - What we know: ONBOARD-10 requires explicit promotion (patch/cherry-pick/push). Phase 8.6 Telegram refactor adds `/commit` and `/approve` commands.
   - What's unclear: Does Phase 8.5 need to expose a promotion endpoint, or just implement the service layer?
   - Recommendation: Implement `PromotionService` with all three strategies and expose `POST /workspaces/runs/:runId/promote` as a REST endpoint. The Telegram command wrapper comes in Phase 8.6.

4. **How does SSH auth testing work on a VPS without interactive prompts?**
   - What we know: `GIT_SSH_COMMAND` can point to a custom script or pass `-i <keyfile> -o StrictHostKeyChecking=no`.
   - What's unclear: Whether `StrictHostKeyChecking=no` is acceptable for MVP or if we need known_hosts management.
   - Recommendation: For MVP, use `StrictHostKeyChecking=accept-new` (accepts new hosts, rejects changed keys). Document this in `docs/SECURITY.md` as a known trade-off. This is standard practice for automated VPS deployments.
